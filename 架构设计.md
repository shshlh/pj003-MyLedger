## 一、项目概述

### 1.1 背景

这是一款个人记账 App，目标是替代随手记。随手记的核心问题是**投资与日常收支数据混在一起**——买基金被记成「支出」，基金涨了被记成「收入」，月度报表完全失真。此外还有清仓账户删不掉、信用卡只能按自然月统计、无周期账单等问题。

### 1.2 核心差异化

| 随手记痛点 | 根本原因 | 本方案 |
|-----------|---------|--------|
| 基金盈亏混入收支报表 | 数据库无投资隔离机制 | `is_investment` 标记字段，投资数据独立统计 |
| 清仓账户无法删除 | 无归档状态设计 | `status` 字段：active / archived / deleted |
| 信用卡按自然月统计 | 无账单周期概念 | `billing_day` + `repayment_day` 字段 |
| 多账户转账需逐笔记 | UI 只支持单笔操作 | 链式转账编辑器 |
| 固定支出需手动记 | 无周期账单功能 | `periodic_bills` 表 + 自动生成引擎 |

### 1.3 设计原则

- **功能切片优先于技术分层**：每个迭代交付一个独立可用的完整体验，不以「先把所有 model 写完」为目标
- **先数据模型后 UI**：表结构设计一次到位（参考随手记踩过的坑），UI 只是数据的消费者
- **本地单设备优先**：先做好本地 SQLite，多设备同步是最后再考虑的事
- **投资标记而非分表**：用 `is_investment` 字段区分交易类型，不建独立的投资交易表，保持 schema 简洁

## 二、架构设计

### 2.1 账本容器架构

随手记最值得借鉴的设计——「以账本为顶层容器」。用户进入一个账本后，所有记账、统计、账户、分类都在该账本下工作：

- **数据天然隔离**：家庭账本和生意账本互不干扰
- **切换方便**：在账本列表快速切换工作空间
- **代码结构一早就支持多账本**：即使目前只需一个，以后扩展不需要重构

```
┌────────────────────────────────────────────┐
│              个人记账APP                     │
│                                              │
│  ┌─────────┐  ┌──────────────────────────┐  │
│  │ 账本列表 │  │      当前账本工作区        │  │
│  │ ──────  │  │  ┌────┬────┬────┬────┐  │  │
│  │ 我的账本◄│  │  │记账│账户│投资│报表│  │  │
│  └─────────┘  │  └────┴────┴────┴────┘  │  │
│               │      │                    │  │
│               │  ┌───┴────────┐           │  │
│               │  │ SQLite 数据库│          │  │
│               │  └────────────┘           │  │
│               └──────────────────────────┘  │
└────────────────────────────────────────────┘
```

### 2.2 模块职责

| 模块 | 文件 | 职责 |
|------|------|------|
| 入口 + 初始化 | `main.dart` | App 启动、数据库初始化、默认账本/账户/分类创建 |
| 首页 | `pages/home_page.dart` | 月收支概览、近期交易、周期账单检查、各 tab 导航 |
| 记账 | `pages/record_page.dart` | 选分类→填金额→选账户→保存，记账后实时刷新首页 |
| 流水 | `pages/transaction_list_page.dart` | 交易记录列表，支持下钻查看 |
| 统计 | `pages/statistics_page.dart` | 月度分类占比 |
| 数据库 | `database/database_helper.dart` | 所有 SQLite 操作：CRUD、记账事务、统计查询、周期账单引擎 |
| 模型 | `models/*.dart` | Book、Account、Category、Transaction、PeriodicBill |

### 2.3 数据流

```
用户操作（记账页）
    ↓
DatabaseHelper.recordTransaction()
    ├── INSERT INTO transactions
    ├── UPDATE accounts SET balance（在事务中）
    └── （投资标记 is_investment 决定该记录进入日常报表 or 投资报表）
    ↓
HomePage / 统计页 → 查询 transactions（按 is_investment 过滤）→ 渲染
```

### 切片 2：账户体系闭环 ✅ 已完成

#### 2.1 信用卡账单周期

**实现要点**：

- 在 `DatabaseHelper` 中新增 `getBillingCycle(Account creditCard, DateTime date)` 方法
- 入参：信用卡账户（含 billing_day 和 repayment_day）+ 任意日期
- 出参：该日期对应的账单周期起止日期
- 规则：
  ```
  账单日每月5日，还款日每月25日
  7月3日消费 → 本期：6月5日 ~ 7月5日，还款日：7月25日
  7月6日消费 → 本期：7月5日 ~ 8月5日，还款日：8月25日
  ```
- 首页新增「信用卡」区域：显示本期已用额度、应还金额、距离还款日天数

**边界测试清单**：

- [ ] 消费日期 = 账单日当天（决定：账单日当天归属本期结束日，即归属本期）
- [ ] 跨年账单周期（12月 → 1月）
- [ ] 月末账单日（31日 → 2月只有28/29天，需取 min(账单日, 当月最大天数)）
- [ ] 非信用卡账户（billing_day 为空）不触发账单周期查询

#### 2.2 链式转账

**实现要点**：

- 在 `record_page.dart` 中新增「链式转账」模式
- 数据结构：
  ```dart
  class TransferNode {
    final String fromAccountId;
    final String toAccountId;
    final double amount;
  }
  ```
- 校验规则：
  - 金额守恒：所有节点的 amount 之和 = 总流出 = 总流入
  - 同一 from 账户的所有转出之和 ≤ 该账户余额
  - 至少包含一个节点（单节点等价于普通转账，应允许）
- UI 交互：
  - 列表形式展示转账节点，每行：from → to, 金额
  - 底部「+」按钮添加节点
  - 提交时在一个事务中执行所有转账

**边界测试清单**：

- [ ] 空链（0 个节点）→ 拒绝
- [ ] 单节点 → 等价于普通转账，应允许
- [ ] 金额不守恒 → 明确报错并标出差异金额
- [ ] 转出总额超过余额 → 明确报错并标出哪个账户不足
- [ ] 自己转自己 → 应拒绝

#### 2.3 账户管理 UI

- 新建 `pages/accounts_page.dart`：账户列表（按类型分组）+ 新建/编辑入口
- 账户编辑：名称、类型、初始余额（仅新建时可设）、账单日/还款日（仅信用卡）
- 归档操作：余额 > 0 时弹窗提示并阻止；余额 = 0 时二次确认
- 已归档账户折叠到「已归档」分组，可恢复

**切片 2 验收标准**：

- 信用卡首页显示本期账单周期和消费金额
- 链式转账一键完成多条记录，每条金额正确，余额同步更新
- 账户列表支持分组展示 + 归档/恢复操作
- `dart analyze` 零错误

### 切片 3：投资中心 ✅ 已完成（2026-07-19）

> **目标**：能买入、跟踪、卖出、清仓、转换基金/股票，盈亏独立统计，不影响日常收支报表。
> 当前完成度：买入/卖出/清仓/转换/净值更新/持仓详情页均已实现。赎回费手动输入。

#### 3.1 投资持仓模型

在 `models/` 下新增 `investment.dart`：

```dart
class InvestmentHolding {
  final String id;
  final String bookId;
  final String accountId;     // 关联的投资账户
  final String code;           // 基金/股票代码
  final String? name;          // 名称
  final String invType;        // fund / stock
  final double totalCost;      // 总成本（累计买入金额 - 累计卖出回款）
  final double totalShares;    // 当前持有份额
  final double? latestNav;     // 最新净值（手动输入 or 联网获取）
  final String? navDate;       // 净值日期
  final String feeType;        // A / B / C / custom（基金费率类型）
  final int isLiquidated;      // 0=持仓中, 1=已清仓
  final String createdAt;
  final String updatedAt;
}
```

**为什么只建一张表**：Kimi 路线图画了 `investments` + `investment_transactions` 两张表。我的判断是不需要第二张——买入/卖出操作直接走 `transactions` 表（`type='invest'`, `is_investment=1`），通过 `related_investment_id` 关联到持仓。这样统计时只需过滤 `is_investment=1`，不用跨表 join。

#### 3.2 买入流程

1. 用户输入：基金代码、买入金额、费率类型、买入日期、扣款账户、当日净值
2. 系统计算（参考 Kimi 的费率表）：
   - 申购费 = 买入金额 × 申购费率
   - 净申购金额 = 买入金额 - 申购费
   - 份额 = 净申购金额 / 当日净值
3. 写入：
   - `transactions`：一条 type=invest, is_investment=1 的记录
   - `investment_holdings`：新建 or 更新持仓（份额累加、成本累加）
   - `accounts`：扣款账户余额 -= 买入金额，投资账户余额 += 买入金额

#### 3.3 卖出与清仓

- 卖出部分份额：输入卖出份额 + 当日净值 → 计算到手金额 − 赎回费 → 更新持仓
- 清仓：卖出全部份额，标记 `is_liquidated=1`
- 清仓后：显示本次投资总盈亏 = 累计卖出回款 − 累计买入成本

#### 3.4 费率参考表（Kimi 贡献）

| 类型 | 申购费 | 赎回费 | 销售服务费 | 适用场景 |
|------|--------|--------|------------|----------|
| A类 | 1.5%（通常打一折 0.15%） | 持有越久越低 | 无 | 长期持有 |
| B类 | 无 | 持有越久越低，最终为 0 | 无 | 中长期 |
| C类 | 无 | 7天内 1.5%，之后 0 | 0.5%/年 | 短期 |
| 自定义 | 用户自行输入 | 用户自行输入 | 用户自行输入 | 特殊情况 |

> 用户可覆盖默认值，支持自定义费率。

#### 3.5 投资盈亏报表

- 新增 `pages/investment_page.dart`：持仓列表（名称/成本/市值/盈亏/收益率）
- 点击进入持仓详情页：交易历史、净值变化、盈亏计算
- 投资盈亏**不进入**首页的月度收支统计（通过 `is_investment=1` 过滤）

**切片 3 边界测试清单**：

- [x] 买入后持仓份额、成本、账户余额正确
- [x] 卖出部分份额后持仓更新正确，到账金额扣除了赎回费
- [ ] 清仓后 `is_liquidated=1`，盈亏计算精确
- [x] 卖出份额超过持仓 → 拒绝并提示
- [x] 投资交易不出现在日常收支报表
- [x] 编辑买入记录不产生幽灵清仓

### 切片 5：自动化与同步

> **目标**：减少手动操作 + 多设备数据互通。**这些是「有更好」，不是「必须有」。**

#### 5.1 净值联网获取

- 方案：先做手动输入（切片 3 已支持），稳定后再探索免费 API
- 候选 API：天天基金、东方财富等公开接口
- 风险：反爬、接口变更，需要容错和降级到手动模式

#### 5.2 CSV 导入/导出

**导出**：统计页 → 导出按钮 → 生成 CSV（含日期/类型/金额/分类/账户/备注）→ 分享/保存

**导入**（支付宝/微信账单）：
```
用户选择 CSV 文件 → 解析 → 字段映射 → 智能匹配分类 → 预览确认 → 批量入账
```

**字段映射参考（Kimi 贡献）**：

| 支付宝字段 | App 字段 | 映射策略 |
|-----------|---------|---------|
| 交易时间 | datetime | 直接映射 |
| 收/支 | type | 「支出」→ expense，「收入」→ income |
| 金额 | amount | 取绝对值 |
| 交易对方 | note | 存入备注，供分类匹配参考 |
| 商品说明 | — | 用于自动匹配分类的关键词识别 |

#### 5.3 WebDAV 同步

```
记账 → 更新 SQLite → 记录变更 → 手动/自动上传到 WebDAV
换设备 → 检查 WebDAV 更新 → 下载 → 覆盖本地（时间戳比对）
```

- 设置页输入 WebDAV 地址/账号/密码
- 手动同步为主，自动同步为可选项
- 冲突策略：时间戳比对，新的覆盖旧的

**切片 5 性能基准**：

- CSV 导出 1000 条记录 < 2 秒
- 数据库备份 < 5 秒
- 启动时间（含周期账单检查）< 2 秒

## 五、数据模型

```sql
CREATE TABLE accounts (
    id TEXT PRIMARY KEY,

    balance REAL NOT NULL DEFAULT 0,
    currency TEXT NOT NULL DEFAULT 'CNY',

    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    FOREIGN KEY (book_id) REFERENCES books(id)
);

CREATE TABLE categories (
    id TEXT PRIMARY KEY,
    book_id TEXT NOT NULL,
    name TEXT NOT NULL,
    type TEXT NOT NULL,            -- expense / income
    parent_id TEXT,                -- 父级分类 ID（二级分类，切片 4 启用）
    icon TEXT,
    sort_order INTEGER NOT NULL DEFAULT 0,
    created_at TEXT NOT NULL,
    FOREIGN KEY (book_id) REFERENCES books(id)
);

CREATE TABLE transactions (
    id TEXT PRIMARY KEY,
    book_id TEXT NOT NULL,
    account_id TEXT NOT NULL,
    to_account_id TEXT,            -- 转账目标账户
    category_id TEXT,
    type TEXT NOT NULL,            -- expense / income / transfer / invest
    amount REAL NOT NULL,
    datetime TEXT NOT NULL,
    note TEXT,
    is_investment INTEGER NOT NULL DEFAULT 0,  -- ⭐ 关键字段：0=日常, 1=投资
    related_investment_id TEXT,    -- 关联的 investment_holdings.id
    created_at TEXT NOT NULL,
    FOREIGN KEY (book_id) REFERENCES books(id),
    FOREIGN KEY (account_id) REFERENCES accounts(id),
    FOREIGN KEY (category_id) REFERENCES categories(id)
);

CREATE TABLE periodic_bills (
    id TEXT PRIMARY KEY,
    book_id TEXT NOT NULL,
    name TEXT NOT NULL,
    type TEXT NOT NULL,            -- expense / income
    amount REAL NOT NULL,
    account_id TEXT NOT NULL,
    category_id TEXT,
    frequency TEXT NOT NULL,       -- daily / weekly / monthly / yearly / custom
    interval_days INTEGER,         -- 自定义间隔天数（frequency=custom 时使用）
    start_date TEXT NOT NULL,
    end_date TEXT,                 -- 空 = 永久
    next_run_date TEXT NOT NULL,
    enabled INTEGER NOT NULL DEFAULT 1,
    created_at TEXT NOT NULL,
    FOREIGN KEY (book_id) REFERENCES books(id),
    FOREIGN KEY (account_id) REFERENCES accounts(id)
);
```

### 5.3 余额变动规则

| 操作 | from 余额 | to 余额 | 统计影响 |
|------|----------|---------|---------|
| 支出 | −amount | — | 进入日常支出报表 |
| 收入 | — | +amount | 进入日常收入报表 |
| 转账 | −amount | +amount | 不影响收支统计 |
| 投资买入 | −amount（资金账户） | +amount（投资账户） | 进入投资报表 |
| 投资卖出 | +amount（资金账户） | −amount（投资账户） | 进入投资报表 |

> 所有余额操作在同一个 SQLite 事务中完成，保证一致性。详见 `DatabaseHelper.recordTransaction()`。

### 5.4 为什么 is_investment 用标记字段而非分表

随手记的教训：投资和收支混在一起是因为当初设计时没有预留隔离机制。Kimi 路线图建议分两张 `investment_transactions` 表，但这样：
- 投资买入/卖出需要同时写两张表 → 增加事务复杂度
- 统计时需要跨表 JOIN → 查询变慢
- 后续加新类型（如「应收应付」）又要加表

用一个 `is_investment` 标记更简洁：所有交易在一张表里，统计时按标记过滤即可。`related_investment_id` 只在需要追溯到具体持仓时使用。
