## 六、开发工作链

### 6.1 分工策略

**主体模式：我亲自开发。** 架构决策、跨模块改动、UI 交互、代码审查、测试——全部由我直接执行。这对一个几千行 Dart 代码的项目来说是最高效的方式，保证风格一致、上下文连贯。

**加速模式：子 agent 做可隔离的机械性任务。** 符合以下条件时用 `codeagent-wrapper --parallel` 分发：

1. 任务有明确输入/输出规格（如「给 account.dart 和 category.dart 写单元测试，覆盖 toMap/fromMap/边界值」）
2. 不依赖运行时状态（不需要启动 App 验证）
3. 不与正在进行的其他改动冲突

**不使用的场景**：
- 涉及 UI 布局和交互的任务（需要视觉判断）
- 需要理解跨模块影响的架构改动
- 任何需要启动 Flutter 验证的任务

### 6.2 单切片工作流

```
1. 读：确认涉及哪些文件，读取当前代码
       ↓
2. 写：逐文件实现（apply_patch），改完一个验证一个
       ↓
3. 审：自查改动（对照审查清单）
       ↓
4. 分析：dart analyze lib\
       ↓
5. 测：运行相关单元/Widget 测试
       ↓
6. 提交：git commit（附改动摘要）
```

### 6.3 代码审查清单（每次提交前过一遍）

**Bug 级别**：
- [ ] async 操作后是否检查了 `mounted`（StatefulWidget 中）
- [ ] 数据库写操作是否在事务中
- [ ] 空值/空列表/边界金额是否有处理

**风险级别**：
- [ ] 新增的数据库查询是否加了 `is_investment=0` 过滤（日常报表）
- [ ] IndexedStack 子页面是否暴露了 `refresh()` 方法
- [ ] 新 model 类的 toMap/fromMap 是否与表结构一致

**风格级别**：
- [ ] 新增页面是否覆盖了加载中/空态/错误态
- [ ] 私有 State 类命名（需要跨文件引用时不加 `_`）
- [ ] 新增文件路径是否符合目录结构约定

### 6.4 测试策略

| 层级 | 范围 | 触发条件 |
|------|------|---------|
| 模型单元测试 | toMap/fromMap、边界值、序列化 | model 类新增或字段变更时 |
| 数据库单元测试 | 增删改查、事务回滚、统计查询 | DatabaseHelper 新增方法时 |
| 业务逻辑测试 | 费率计算、账单周期、余额变动规则 | 新增计算逻辑时 |
| Widget 测试 | 关键交互（记账按钮、金额输入） | 页面大改时 |
| 边界测试 | 按切片中的边界测试清单执行 | 切片完成时 |

> 测试不追求覆盖率数字（90% 对个人项目不现实），追求「所有边界条件和人容易犯错的路径都有测试」。

### E. v0.3 开发反馈（2026-07-15）

**切片 2 踩坑记录**：

| 问题 | 根因 | 避免方式 |
|------|------|---------|
| 信用卡消费余额更新方向反了 | 统一用 balance - amount，但信用卡余额表示债务，消费应增加余额 | 账户类型感知：信用卡支出 balance + amount，其他 balance - amount。每次新增交易类型先想清楚余额含义 |
| 信用卡账单周期逻辑错误 | 只算了当前周期的在途消费，没考虑上期应还的概念 | 信用卡有两个概念：当前周期（在途，未出账单）和上期（已出账单，要还的）。两者分开计算 |
| 还款功能方向搞反 | 偿还信用卡时债务减少，但 transfer 统一 balance + amount | 转账到信用卡时做 balance - amount（债务减少）。所有 to-account 的余额变更都要先查账户类型 |
| 转账/链式转账崩溃 | _toAccountId 可能等于 _accountId，下拉列表过滤后列表为空 | 所有 _firstToAccountId() 调用加 exclude: _accountId 参数。链式节点 _addChainNode() 额外检查 toId != fromId |
| PowerShell 正则替换破坏文件结构 | 用 -replace 或 Regex.Replace 跨行匹配时，匹配边界不精确导致误伤 | 能用 apply_patch 就别用正则。需要大量替换时写 Python 脚本到文件再执行 |
| Git checkout 丢失未提交修改 | 为修复正则损坏的文件执行了 git checkout -- file，包含了全部未提交的工作 | git checkout -- file 前先用 git diff 确认这文件真有提交版本可回退 |
| DropdownButtonFormField.value 弃用 | Flutter 3.33+ 弃用了 value 参数 | 改用 initialValue 配合 ValueKey 强制重建 |
| apply_patch 不支持一个 patch 改多个文件 | patch 语法限制 | 一个 apply_patch 调用只改一个文件，多文件改动并行发多个调用 |
| 正则匹配了多个 }); | 想修复一个合并问题，结果匹配了文件中所有同样格式的行 | 修改前先确认匹配的唯一性，或使用更精确的上下文锚定 |

**v0.3 核心经验总结**：

1. 账户类型感知是系统性风险 - 余额变更逻辑不能一刀切。储蓄卡、信用卡、投资账户的余额含义不同 |
2. 信用卡不是普通账户 - 它有两个时间线（当前周期和已出账单），UI 上要分开展示在途和应还 |
3. 严格的分步工作流 - 每个功能切片有明确的验收标准。在验收通过之前，不继续下一项 |
4. 正则文件替换是最后手段 - 它不可逆、不可调试。apply_patch 更安全，Python 脚本次之 |
5. 子 agent 用得少 - 这个项目规模下，亲自开发比分发子任务更高效


### F. v0.4 代码审查踩坑记录（2026-07-18）

> 本节记录 2026-07-17 ~ 07-18 三轮审查中发现的真实 bug 和逻辑错误，按严重程度排列。

**审查范围**：全部 Dart 源码（约 2500 行）+ 数据库 schema + 设计文档。

#### 阻断级（P0）—— 会导致 INSERT 失败或数据静默损坏

| 问题 | 根因 | 修复方式 |
|------|------|---------|
| deleteTransaction() 方向判定永远不成立 | isFromInvest = fromRows.first['type'] == 'invest'，但 accounts.type 只有 fund/stock，没有 invest | 改为 fund || stock |
| recordChainTransfer() 缺 updated_at | transactions 表 DDL 有 updated_at TEXT NOT NULL，但硬编码 INSERT map 没写这个字段 | 补 updated_at: now |
| periodic_bills 表缺 updated_at 列 | DDL 没有定义该列，但 runDueBills() 尝试写入 updated_at | 加列 + DB version 3->4 + _upgradeDb 迁移 |
| Transaction model 缺 updatedAt 字段 | toMap() 和 fromMap() 不含 updated_at，早期 INSERT 全靠硬编码 | 补字段 + 升级 DDL 同步 |

#### 高优先级（P1）—— 逻辑错误，日常使用中会触发

| 问题 | 根因 | 修复方式 |
|------|------|---------|
| 编辑投资记录始终走买入路径 | _editTransaction() 中不管 note 是"赎回本金"还是"转换转出"，都调 recordInvestment()（买入） | 按 note 第一词区分：只有"买入"才走买入重建，其他跳过并提示 |
| 同一持仓多次卖出删除时误删盈亏记录 | P&L 查询用 related_investment_id 匹配，无法区分不同笔次的卖出 | 用 batch_id 精确匹配（sellInvestment 已生成 batchId） |
| 买入删除后投资账户余额有误差 | recordInvestment() 末尾做了市值重算（覆盖为份额*净值），但 deleteTransaction() 买入分支没重算 | 买入删除分支追加市值重算（已实现） |
| switchFund() 手续费/退回成为孤立数据 | 手续费和退回记录设了 related_investment_id 但 category_id=null，P&L 查询条件 category_id IS NOT NULL 匹配不到 | deleteTransaction 改用 batch_id 匹配（已实现） |

#### 中等（P2）—— 不立刻炸但会累积问题

| 问题 | 根因 | 修复方式 |
|------|------|---------|
| runDueBills() 无去重 | 一天内多次重启 App 且 next_run_date 更新失败时，同一笔周期账单重复生成 | 生成前检查今天是否已有同 book_id+account_id+amount+note 的记录 |
| 删除买入时按金额比例估算份额不精确 | sharesToRemove = oldShares * (amount / oldCost)，多次买入且净值不同时有偏差 | 优先从 note 中解析份额精确值，失败再 fallback |
| 流水页无日期过滤 | getTransactions() 返回全部记录，上千条后性能下降 | 加 startDate 参数，默认只加载最近 90 天 |
| InvestmentHolding.total_cost 语义模糊 | 经历了买入累加和卖出按比例扣减后，实际是"剩余成本基数"，不是"累计投入总额" | 加注释澄清 |

#### UX/规范（P3）

| 问题 | 修复 |
|------|------|
| 颜色惯例反了（支出=绿、收入=红） | 改为支出=红、收入=绿，符合中国记账习惯 |
| 测试覆盖不足 | 补充关键路径的数据库集成测试（本轮增加） |

#### 本轮审查核心经验

1. **审查报告过时风险**——代码在持续修改，审查报告生成时的结论可能已不适用。审查前先用 grep 确认问题代码是否已被修复，避免做重复工作。
2. **DELETE 方向判定是系统性风险**——余额回滚方向错了数据就静默损坏，没有报错。每次新增交易类型必须同时 review deleteTransaction 和 recordTransaction 的余额方向。
3. **投资记录有三种类型**（买入、卖出本金、盈亏损益），不能只用 type == 'invest' 判断，必须结合 note 内容区分。编辑操作尤其容易踩坑。
4. **batch_id 是投资记录删除的正确模式**——用关联字段跨表匹配盈亏记录时，batch_id（单次操作批次）比 related_investment_id（持有期标识）更精确，不会被多次操作干扰。
5. **DB 迁移要提前设计好**——prod 环境已有 DB 时，DDL 改动必须配 _upgradeDb 迁移。SQLite ALTER TABLE 有限制（只能 ADD COLUMN，不能改 NOT NULL），设计初始 schema 时要留余量。

#### v0.5 做得好

1. **Delta 更新模式** — `updateTransactionAmount` / `updateInvestmentBuy` 不删重建，按差值调余额和持仓。编辑 N 次不留垃圾。
2. **银行流水对账设计** — 储蓄卡只显示银行实际发生的流水（净到账），手续费和投资收益在投资账户内展示，储蓄卡对得上银行账单。
3. **手动输入费率** — 赎回费由用户手动填，不搞 FIFO 逐笔匹配计算，大幅降低系统复杂度。
4. **投资账户作为虚拟账户的理解** — 投资账户（天天基金）不是银行账户，它的余额代表市值，流水可以包含各种投资操作（买入/卖出/转换/手续费），不影响真实银行账户的核对。
5. **测试覆盖从 9 到 28** — 数据库关键路径全部有集成测试兜底，错误率大幅下降。
